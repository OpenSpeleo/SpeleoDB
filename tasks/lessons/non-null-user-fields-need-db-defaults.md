# Non-Null User Fields Need Database Defaults

## Trigger

Adding `User.has_api_doc_access` as a non-null Boolean with only a Python
default broke `test_landmark_collection_migration_rollback_restores_landmark_user`.

That test migrates the `gis` app backward and creates a `users.User` through a
historical model state that predates newer user fields. The live database table
still contains the newer non-null column, but the historical ORM insert omits
that column, causing a database-level `NOT NULL` failure.

## Rule

When adding a non-null field to a central model that historical migration tests
may insert through older app states, provide both:

- `default=...` for normal ORM/model behavior
- `db_default=...` for database inserts generated by historical model states

This is especially important for `users.User`, because unrelated app migration
tests can create historical users while the physical table remains at the
latest schema.

## Fix Pattern

```python
has_feature_access = models.BooleanField(
    default=False,
    db_default=False,
)
```

The migration must include the same `db_default` so the database schema has a
real default for omitted historical columns.
